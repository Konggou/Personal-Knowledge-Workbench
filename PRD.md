# 个人知识工作台 v1 PRD

## 1. 产品定义

个人知识工作台 v1 是一个面向单用户、Windows 优先、本地运行、聊天优先的知识工作产品。

它的核心不是“泛聊天”，而是：

- 用 `项目` 组织资料
- 用 `会话` 承载项目内持续对话
- 用 `消息` 承载问答、调研状态和结果沉淀
- 用 `来源` 保证回答可追溯
- 用 `知识库` 统一管理网页和文件资料

它不是：

- 面试产品
- 多人协作系统
- 企业知识管理平台
- 纯 Agent 平台
- 普通无来源聊天壳

## 2. 目标用户

目标用户是普通知识工作者。

典型场景：

- 围绕一个持续主题收集网页、PDF、DOCX
- 在自己的资料库里提问
- 在复杂问题上升级为深度调研
- 将高价值结论沉淀为摘要或报告

## 3. 核心问题

用户真正的问题不是“缺一个聊天框”，而是：

- 资料分散，缺少项目级知识容器
- 只能问，不能稳定基于自己的资料回答
- 回答没有来源边界，难以复核
- 复杂问题需要多步整理，人工成本高
- 好结果难以沉淀和复用

## 4. 核心价值

v1 只聚焦三件事：

1. 围绕项目管理资料
2. 围绕资料进行可信问答
3. 在复杂问题时升级为调研并沉淀结果

## 5. v1 范围

### 5.1 核心对象

- 项目
- 会话
- 消息
- 资料
- 来源
- 摘要卡
- 报告卡

### 5.2 核心用户动作

对用户暴露的主要动作只有：

- 提问
- 深度调研
- 生成报告

补充动作：

- 保存为摘要
- 增加资料
- 查看来源

### 5.3 绝对主闭环

`创建项目 -> 新建会话 -> 增加资料 -> 项目内提问 -> 查看来源`

扩展闭环：

`复杂提问 -> 升级为调研 -> 保存为摘要 / 生成报告`

## 6. 信息架构

一级导航固定为：

- `工作台`
- `会话`
- `知识库`

路由固定为：

- `/workspace`
- `/sessions`
- `/knowledge`
- `/projects/[projectId]`
- `/projects/[projectId]?sessionId=...`

以下旧结构不再属于 v1：

- `/tasks`
- `/search`
- `/assets`
- 独立 task 详情页

## 7. 产品心智

### 7.1 项目

项目是知识库容器，负责：

- 承载资料
- 定义资料的检索范围
- 承载多个会话

### 7.2 会话

会话是项目内对话线程。

一个项目可以有多个会话，但这些会话共享同一个项目知识库。

新会话意味着：

- 清空聊天上下文
- 继续使用同一个项目资料库

### 7.3 消息

消息是主交互单元，包含：

- 用户提问
- 助手回答
- 状态卡
- 来源更新卡
- 摘要卡
- 报告卡

### 7.4 来源

来源是回答底部可展开的轻量证据层：

1. 回答底部显示 `n 个来源`
2. 点击后先展开来源标题列表
3. 再点击标题，打开详细预览

## 8. 关键功能定义

### 8.1 工作台

工作台是唯一项目入口页。

必须包含：

- 创建项目表单
- 最近活跃项目列表
- 项目名搜索
- 归档项目默认隐藏

完成定义：

- 用户能创建项目
- 项目按最近活跃排序
- 点击项目进入项目聊天页

### 8.2 项目聊天页

项目聊天页是 v1 主工作区。

结构固定为：

- 顶部轻导航
- 左侧整体化 sidebar
- 中间主聊天区

不再保留：

- 项目页大型头部
- 常驻右侧知识库栏
- 多卡片并排工作台结构

完成定义：

- 无 `sessionId` 时不自动打开任何会话
- 主区显示项目空态
- 左侧展示所有项目树
- 当前项目默认展开，其它项目默认收起
- 左侧收起后主聊天区真实变宽

### 8.3 会话命名

- 会话创建时默认无正式标题
- 用户发出首条消息后自动生成标题
- 用户允许手动改名

### 8.4 空项目与弱资料模式

规则固定为：

- 没有会话时，不显示聊天输入区
- 先让用户“新建会话”或“去知识库添加资料”
- 一旦已经创建会话，即使没有资料，也允许继续聊天
- 这种场景必须明确标记为“弱资料模式”

### 8.5 输入区

输入区必须包含：

- `增加资料`
- 文本输入
- `深度调研`
- `生成报告`
- `发送`

其中：

- 输入框固定在聊天区底部
- 输入框与消息列共用中轴和宽度
- 所有动作都是轻量小胶囊，不是大按钮卡片
- `增加资料` 点击后弹出极小菜单：
  - 添加文件
  - 添加网页链接
- `深度调研` 是一次性开关，仅影响本次发送，发送后自动复位
- `生成报告` 默认基于当前会话最近一次有效结论

### 8.6 增加资料

资料添加有两条路径：

- 会话内添加
- 知识库页添加

规则固定为：

- 没有会话时，不提供会话内添加
- 有会话后，即使这条会话还是空白，也允许会话内添加
- 添加成功后，会话中插入一条轻量系统卡

### 8.7 提问与调研

普通提问默认走项目内问答。

命中资料后的普通 grounded 回答主链固定为：

- 混合检索召回候选证据
- 按规则决定是否 rerank
- 选出最终少量证据
- 基于最终证据流式生成回答
- 完成后再挂载最终来源气泡

普通问答不会因为问题复杂而自动升级为调研。

只有用户主动开启 `深度调研` 时才升级为调研。

升级后规则：

- 保留在同一会话
- 不跳独立任务页
- 用轻量状态卡表达，如 `调研中`、`调研完成`

### 8.8 来源层

每条回答底部显示来源气泡。

规则固定：

- 文案是 `n 个来源`
- 最多显示 3 个图标
- 只展示真正喂给模型的最终证据，不展示召回候选
- 第一层只显示来源标题，不显示摘录
- 详细来源预览采用覆盖式浮层，不重新引入常驻右栏

### 8.9 摘要卡与报告卡

结果沉淀不再使用独立资产页，而是沉淀在当前会话里。

规则固定：

- 保存为摘要后追加一张摘要卡
- 生成报告后追加一张报告卡
- 不覆盖原始回答
- 结果卡只保留 `复制 / 删除`
- 删除是软删除，前台不提供恢复

### 8.10 会话页

会话页按项目分组展示会话。

完成定义：

- 只做浏览，不做全局搜索
- 支持进入项目聊天页
- 支持重命名和删除

### 8.11 知识库页

知识库页是资料管理主入口。

必须包含：

- 全局搜索资料
- 按项目筛选
- 按项目分组浏览
- 每个项目分组头部支持增加资料
- 先预览，再进入聊天

## 9. 成功标准

1. 用户打开项目页后，第一感知是“我在一个整体聊天界面里工作”，而不是“我在操作多个并排卡片”。
2. 用户能在项目内顺畅完成“提问 -> 看来源 -> 继续追问”。
3. 用户能在需要时切换深度调研并沉淀摘要或报告。

## 10. 非目标

v1 明确不做：

- 多人协作
- 认证/权限
- workspace 抽象
- 独立 task dashboard
- 独立 asset center
- 独立 search page
- 正式插件系统
- 桌面壳分发
